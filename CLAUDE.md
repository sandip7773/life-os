# Life OS — project context for Claude

## What this is
A personal life management system I'm building solo. A Telegram bot is
the unified interface; different life domains (Health, Money, Career,
Mentorship) are handled by separate Python modules behind a thin
orchestrator. Storage is Supabase (Postgres).

## Design philosophy
- Motivation over elegance. I have a history of stalling on projects
  like this, so early visible wins are a design requirement, not a
  nice-to-have.
- Local first, deploy later.
- Each day should end on a commit and a working thing.
- Push back on unnecessary complexity. If I'm about to build something
  that isn't needed today, call it out.

## Architecture rules
- Orchestrator is a thin dispatch layer. No domain logic there.
  Routing: slash commands first, LLM classifier second, clarify
  fallback third.
- `shared/` holds the LLM client and Supabase client, wrapped once.
  Modules never talk to APIs directly.
- Two schema shapes, not interchangeable:
    - generated plan content = JSON/text blobs
    - logged entries = structured rows with numeric precision
- `numeric(12,2)` for all financial data, never float.
- Cheap/fast models (GPT-4o-mini, Gemini Flash) for high-volume
  classification. Claude for nuanced reasoning (plan generation,
  meal parsing where it matters).

## Stack
- Python 3, WSL (Ubuntu on Windows).
- Telegram bot via `python-telegram-bot`.
- Supabase (free tier, ap-southeast-2), not yet live in code.
- `anthropic` SDK for Claude, will add OpenAI/Gemini later for
  classification.
- n8n reserved for scheduled/triggered automation, separate from
  conversational logic.
- Project structure: `life-os/{bot,modules/{health,money,career,
  mentorship},shared,db,dashboard,docs,scripts}`.
